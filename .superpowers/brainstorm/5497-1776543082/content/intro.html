Making Gyst Self-Documenting

Brainstorming: How should teams structure KB entries so AI agents never need to read source code?

The Problem

When an AI agent calls recall("how does the knowledge graph work") in a gyst-powered codebase, it should get a complete answer — not silence that forces it to read 5 source files.

Today, even gyst itself doesn't document its own architecture in its KB. The fix isn't just writing more entries — it's building a pattern that any team can follow.

What We're Designing

  • What AI agents actually look for when they call recall()
  • Entry structure (titles, content, tags, graph links) that maximizes hit rate
  • Patterns: ghost knowledge, module summaries, Q&A entries, graph connectivity
  • A gyst self-document command or workflow to generate/maintain these entries

Scope (User chose B)

Build a reusable pattern — gyst becomes the reference implementation. Any team adopting gyst can run one command and get a self-documenting codebase.
