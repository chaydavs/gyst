Design: Section 1 — Architecture Overview

Four layers working together so agents never read source files

The 4-Layer Self-Documenting KB
Layer 0 — Ghost Knowledge (always surfaces, infinite confidence)

Auto-generated from Graphify: top 10 most graph-central nodes become ghost entries. "How does the search pipeline work?", "What are the three graph layers?", "How does RRF fusion work?" — guaranteed tier 0 retrieval. Updated on structural change detection.

Layer 1 — Structural Skeleton (Graphify → KB promotion)

Every module/file in the codebase gets a KB entry auto-derived from its exports, imports, and git blame. Zero LLM tokens. Runs on every git commit. Entries link to each other via call-graph edges — traversal from any entry reaches the full module context.

Layer 2 — MD File Corpus (plans, specs, ADRs, docs)

All markdown files auto-ingested: README, ARCHITECTURE, decisions/*, docs/**, CLAUDE.md, superpowers/specs/**. Stored with source_file pointer. Re-ingested on file change (hash check). Shown in dashboard "Docs" tab. Agents retrieve full content + surrounding KB entries in one recall().

Layer 3 — Harvest Corpus (WHY, decisions, failure patterns)

Extracted from agent sessions (PostToolUse + Stop), git commit messages (post-commit hook), and PR descriptions. Confidence-scored, decay-eligible. These are the entries that capture intent: why a decision was made, what failed, what the tradeoff was. Self-populates as the team works.

How a recall() call traverses all 4 layers

  1. Ghost entries (Layer 0) always pre-loaded in tier 0
  2. BM25 + RRF fusion hits Layers 1, 2, and 3 simultaneously
  3. Graph traversal walks edges from any match into related modules
  4. Agent gets: structural context + MD docs + session learnings in one response
