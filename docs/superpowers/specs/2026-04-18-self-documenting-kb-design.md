# Self-Documenting Knowledge Base Design

**Date:** 2026-04-18
**Status:** Approved
**Scope:** V1 implementation — gyst as reference implementation for any team

---

## Problem

When an AI agent works in a gyst-powered codebase, it should be able to call `recall("how does X work")` and get a complete answer. Today this fails in four ways:

1. No entry exists for the concept (KB miss → zero results → retry loop)
2. The right entry exists but ranks below noise (BM25 miss → agent falls back to source reads)
3. Agent never calls recall() — doesn't trust a sparse KB (cold-start problem)
4. Agent needs multiple recall() calls to piece together full context (high token cost)

Each failure wastes tokens and produces worse output. The fix is not writing more entries manually — it's building a system that makes the KB self-populate and stay accurate automatically.

---

## Design Goal

Any team that installs gyst and runs `gyst self-document` gets a KB that AI agents can query without ever reading source files. No manual entry writing. Zero ongoing maintenance cost.

---

## Architecture: 4-Layer KB

### Layer 0 — Ghost Knowledge (guaranteed tier 0 retrieval)

Top 10 most graph-central concepts become ghost knowledge entries. Ghost entries have infinite confidence and always surface in recall() tier 0 regardless of BM25 score.

- Generated by `gyst self-document` Phase 3
- Input: structural entry + linked MD docs + git blame summary
- One Haiku call per entry (~500 tokens each, ~$0.001 total for 10 entries)
- Regenerated when centrality ranking changes: compare stored centrality scores to current graph — if top-10 set changes, regenerate affected ghost entries only
- Examples: "How does the search pipeline work?", "What are the three graph layers?", "How does RRF fusion work?"

### Layer 1 — Structural Skeleton (zero tokens, runs on every commit)

Every module and file in the codebase → KB entry auto-derived from exports, imports, and call graph. Generated by promoting Graphify AST nodes to KB entries.

- Type: `structural`
- Confidence: 0.8 (structural entries don't decay)
- Graph edges: call-graph relationships from Graphify
- Trigger: post-commit git hook, `gyst self-document` Phase 1
- No LLM involved — pure AST analysis

### Layer 2 — MD Document Corpus (auto-ingested, hash-checked)

All markdown files in the codebase are automatically ingested as KB entries with `source_file` pointers. Shown in dashboard Docs tab.

**Files ingested:**
- `README.md`, `CLAUDE.md`, `ARCHITECTURE.md`
- `docs/**/*.md`
- `decisions/**/*.md` (ADRs)
- `docs/superpowers/specs/**/*.md`
- `.claude/rules/**/*.md`
- Any `*.md` file at repo root

**Process per file:**
1. Compute SHA hash → skip if unchanged vs `source_file_hash` in DB
2. Parse frontmatter for tags and title
3. Extract H2/H3 headings → section summaries
4. Named entity extraction → graph edges to structural nodes
5. Store as `md_doc` type, confidence 0.9, with `source_file` path

**Triggers:**
- `FileChanged` hook on `**/*.md` → immediate reingest on save
- `InstructionsLoaded` hook → auto-ingest CLAUDE.md and rules files on session start
- `gyst self-document` Phase 2 — full glob scan
- `git pull` hook — rescan for changed files

### Layer 3 — Harvest Corpus (self-populates from agent work)

Captured from: agent sessions, git commit messages, and tool failures. These entries contain the WHY — decisions, patterns, failure modes — that source code can't express.

**Harvest triggers:**
- `PostToolUse` — extract learnings from every significant tool call
- `PreCompact` — full transcript harvest before context compaction (highest value trigger)
- `PostCompact` — verify harvest landed, take drift snapshot
- `Stop` / `SubagentStop` — session distillation
- `PostToolUseFailure` — auto-extract error_pattern entries from real failures
- `git post-commit` hook — extract learning from commit message + diff summary

## Hook Expansion: 6 → 12

### New hooks added

| Hook | Script | What it captures |
|------|--------|-----------------|
| `PreCompact` | `pre-compact.js` | Full transcript harvest before context is erased |
| `PostCompact` | `post-compact.js` | Verify harvest success, take drift snapshot |
| `InstructionsLoaded` | `instructions-loaded.js` | Auto-ingest CLAUDE.md and rules files |
| `FileChanged` (*.md) | `file-changed.js` | Auto-reingest any MD file on save |
| `PostToolUseFailure` | `tool-failure.js` | Auto-extract error_pattern from real failures |
| `SubagentStart` | `subagent-start.js` | Inject ghost knowledge into every subagent context |

### PreToolUse(Read) — KB miss tracking

When an agent reads a source file via the `Read` tool, that's a KB miss signal. The PreToolUse hook logs: which file was read, what the preceding prompt was (from `transcript_path`). These files are prioritized in the next `gyst self-document` run.

This makes the KB miss rate measurable and feeds directly into the drift score.

| Structural | Blue `#2563eb` |
| File / Function | Indigo `#6366f1` |

### Node sizing
- Node radius = `base_radius + (connection_count * scale_factor)`
- Ghost entries always render at max size regardless of connections
- Minimum radius: 3px, maximum: 20px

### Edge rendering
- Edge thickness = co-retrieval strength (thin = structural link, thick = frequently co-retrieved)
- Edges only render when connection count > 0 (current: 0 edges because no co-retrievals yet)

## gyst self-document Command

```
gyst self-document [--project-dir <path>] [--force] [--ghost-count <n>]
```

**Phase 1 — Structural Skeleton** (~2s, zero tokens)
- Run Graphify AST on entire codebase
- Promote top-level modules and key functions to KB entries (type: `structural`)
- Build call-graph edges in `relationships` table

**Phase 2 — MD Corpus Scan** (~1s, zero tokens)
- Glob all `*.md` files
- Hash-check each against stored `source_file_hash`
- Ingest changed/new files only
- Link to structural nodes via entity extraction

**Phase 3 — Ghost Knowledge Generation** (~5000 tokens Haiku, ~$0.001)
- Compute betweenness centrality on knowledge graph
- Select top N nodes (default: 10, configurable via `--ghost-count`)
- For each: one Haiku call with structural entry + linked MD docs as context
- Write ghost knowledge entry with infinite confidence

**Output:**
```
gyst self-document complete
  Structural entries: 147 created, 23 updated
  MD documents: 12 ingested, 3 skipped (unchanged)
  Ghost entries: 10 written
  Tokens used: 4,823 (Phase 3 only)
  KB miss queue: 0 files pending
```

## Dashboard: Docs Tab

New tab in the dashboard sidebar between "Review" and "Team":

- **File list** — all ingested MD files, sorted by last modified
- **Preview pane** — full rendered markdown on click
- **Linked entries panel** — KB entries linked to this doc (right sidebar)
- **Graph node** — cyan node in graph view, click opens doc in Docs tab

Agents can surface doc content via `recall()` — responses include `source_file` path + the most relevant excerpt (max 500 tokens).

## Token Budget Impact

| Scenario (before) | Token cost |
|-------------------|-----------|
| Agent reads 3 source files to answer arch question | ~6,000 tokens |
| Agent makes 4 recall() calls finding nothing | ~400 tokens wasted |

| Scenario (after) | Token cost |
|-----------------|-----------|
| Agent calls recall() once, gets ghost + MD excerpt | ~800 tokens |
| Bootstrap (one-time self-document) | ~5,000 tokens |

**Net reduction: ~87% per architectural query after bootstrap.**
